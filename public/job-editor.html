Paso 1: Fuente de datos

Identifica unívocamente cada registro. Se usa para detectar cambios y eliminar duplicados. Ej: Id, codigo.

Incremental: solo trae registros modificados desde la última ejecución (requiere campo de fecha). Full: lee toda la tabla cada vez.

Campo datetime que indica la última modificación del registro. Sin él, el incremental lee todo igual que Full.

Categoría del objeto en ArdisApp. Determina cómo procesa cada ítem la API destino.

Snapshot: el sistema detecta automáticamente creaciones, modificaciones y bajas comparando con la ejecución anterior. Passthrough: cada registro trae el tipo de operación en un campo propio.

Cada fila debe tener este campo con valor: upsert (crear o actualizar) o delete (eliminar).

Si está activo, se hace el request a la API incluso cuando no hay registros nuevos o modificados. Útil para APIs que esperan un heartbeat periódico.

Paso 3: Mapeo de campos

Definí cómo se construye el objeto que se envía a la API. Cada fila es un campo del JSON de salida. Elegí el modo según la fuente del valor:

Campo Fuente

Toma el valor directamente de una columna del origen de datos. Podés aplicar una transformación (mayúsculas, número, fecha ISO, etc.).

Ej: columna Descripcion → campo API nombre

Fijo

Asigna un valor literal constante al campo API. Siempre envía el mismo valor, independientemente de los datos del origen.

Ej: espesor = 10 — o activo = true

Expresión

Calcula el valor mediante una condición. El builder visual arma un ternario simple; la expresión avanzada permite JS completo con todas las columnas como variables.

Ej: ESPESOR >= 18 ? "Grueso" : "Fino"

Click en un campo para agregarlo al mapeo. Los campos ya mapeados aparecen marcados.

Campo API Modo Fuente / Valor Transform

El filtro se evalúa fila por fila antes de armar el JSON de salida. Las filas que no cumplan la condición se saltan — no se envían a la API.

Cada columna del origen SQL está disponible como variable con su nombre exacto. Si el valor es texto, comparalo entre comillas: CATEGORIA == "Sheet". Si es número, sin comillas: ESPESOR > 4.

Referencia de expresiones de filtro JS · fila por fila

Cómo funciona

El filtro es una expresión JavaScript que se evalúa una vez por cada fila del origen SQL. Todas las columnas de esa fila están disponibles como variables locales con su nombre exacto. Si la expresión devuelve true → la fila pasa al envío. Si devuelve false → se omite.

// columnas disponibles como variables:
SheetExt01, ESPESOR, CATEGORIA, PRECIO ...

Tipos de datos en SQL Server → JavaScript

Tipo SQL Llega como Ejemplo de valor Cómo comparar
nvarchar / varchar string "Roble" CAMPO == "Roble"
int / decimal / float number 18 ESPESOR >= 18
bit 1 / 0 o true/false 1 !!ACTIVO
NULL (cualquier tipo) null null CAMPO != null
datetime / date Date o string "2024-01-15" String(FECHA).includes("2024")

Problema frecuente: campos "vacíos" en SQL Server

Un campo puede parecer vacío de varias formas. Hay que cubrirlas todas:

Lo que ves en el grid Lo que llega en JS Lo captura != null?
NULL null ✓ sí
(celda vacía) "" ✗ no — es string vacío
"NULL" (texto literal) "NULL" ✗ no — es string
0 0 ✗ no — es número 0
"0" (texto) "0" ✗ no — es string "0"
" " (espacios) " " ✗ no — trim() necesario

Expresión recomendada para "campo tiene valor real":

SheetExt01 != null && String(SheetExt01).trim() !== "" && String(SheetExt01).trim() !== "0" && String(SheetExt01).trim().toUpperCase() !== "NULL"

Funciones útiles

String(CAMPO) convierte a string — seguro con null/number/boolean
String(CAMPO).trim() elimina espacios al inicio y al final
String(CAMPO).toUpperCase() convierte a mayúsculas para comparar sin importar el case
Number(CAMPO) convierte a número — si el campo llega como string "18"
isNaN(Number(CAMPO)) true si el valor no es un número válido
!!CAMPO convierte a boolean — false para 0, null, "", undefined
/patrón/.test(String(C)) regex — ej: /^MAT-\d+$/.test(String(CODIGO))
["A","B"].includes(CAMPO) equivalente a SQL IN — campo debe ser uno de los valores

Ejemplos por caso de uso

Campo con valor real (sin NULL / vacíos / ceros)

SheetExt01 != null && String(SheetExt01).trim() !== "" && String(SheetExt01).trim() !== "0" && String(SheetExt01).trim().toUpperCase() !== "NULL"

Solo registros de una categoría

CATEGORIA == "Sheet"

Rango numérico

ESPESOR >= 10 && ESPESOR <= 30

Múltiples categorías (equivalente a IN)

["Sheet", "Panel", "MDF"].includes(String(CATEGORIA))

Nombre contiene texto (case insensitive)

String(DESCRIPCION).toUpperCase().includes("ROBLE")

Código con formato específico (regex)

/^MAT-\d+$/.test(String(CODIGO))

Registro activo con precio positivo

!!ACTIVO && Number(PRECIO) > 0

Campo presente Y categoría específica (combinado)

SheetExt01 != null && String(SheetExt01).trim() !== "" && String(SheetExt01).trim().toUpperCase() !== "NULL" && CATEGORIA == "Sheet"

Excluir registros de prueba

!String(DESCRIPCION).toUpperCase().startsWith("TEST") && !String(DESCRIPCION).toUpperCase().startsWith("PRUEBA")

Comportamiento ante errores

Si la expresión lanza un error de JavaScript (variable no definida, sintaxis incorrecta, etc.), la fila se incluye por defecto — no se omite. El error se registra en los logs de ejecución del job. Para debuggear, revisá los logs después de ejecutar manualmente.
